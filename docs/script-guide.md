# 剧本编写指南

这份文档面向剧本作者，目标是帮助你写出一份可以直接导入到 Script Simulator 中运行的剧本 JSON。

文档内容会严格对应当前项目实现，而不是理想中的未来设计。换句话说，你照着本文档写，剧本应该能被当前版本正确读取、校验和运行。

## 1. 先理解这个系统在做什么

这个项目不是传统分支小说，也不是纯文字 RAG 聊天。它的运行方式更接近：

1. 你定义一套角色、维度、事件、阶段、结局和开局选项。
2. 玩家进入剧本，先完成开局选项。
3. 系统把当前设定、最近对话、数值状态、阶段信息一起交给 LLM。
4. LLM 返回一轮结构化输出：
   - 剧情正文 `narrative`
   - 候选选项 `options`
   - 数值更新 `values`
   - 可选关键事件标记 `keyEvent`
5. 系统根据你的事件条件、阶段条件、结局条件继续推进状态。

所以，一份好剧本不是“写很多剧情文本”，而是“把剧情生成的边界、触发器和节奏写清楚”。

## 2. 一份剧本的最小结构

一份可运行剧本本质上是一个 JSON 对象。

最小可导入示例：

```json
{
  "name": "未命名剧本",
  "description": "一句话介绍这个剧本",
  "characters": [
    { "id": "player", "name": "玩家", "description": "玩家角色" }
  ]
}
```

这份剧本虽然很简陋，但已经满足最基本要求：

- 顶层是对象
- 有 `name`
- `characters` 中包含 `id: "player"` 的角色

但它几乎没有可玩性。真正建议的完整结构是：

```json
{
  "name": "剧本名",
  "description": "简介",
  "rules": {
    "writing_style": "文风要求",
    "forbidden": ["禁止事项1", "禁止事项2"],
    "requirements": ["必须做到1", "必须做到2"]
  },
  "dimensions": [],
  "characters": [],
  "events": [],
  "stages": [],
  "endings": [],
  "setup": []
}
```

## 3. 顶层字段说明

### `name`

- 类型：字符串
- 作用：首页展示名、编辑器标题、会话归属
- 建议：直接写玩家能理解的剧本名，不要过长

示例：

```json
"name": "午夜列车调查"
```

### `description`

- 类型：字符串
- 作用：首页摘要、作者自己做内容定位
- 建议：用 1 到 3 句话说明题材、核心冲突、体验风格

示例：

```json
"description": "一场发生在跨城夜车上的悬疑互动，玩家需要在多名角色之间判断真相。"
```

### `cover`

- 类型：字符串
- 当前状态：字段会被保留，但当前前端没有正式使用
- 建议：可以不写

## 4. 写作规则 `rules`

`rules` 用来控制 LLM 的语气、边界和输出倾向。

结构：

```json
"rules": {
  "writing_style": "……",
  "forbidden": ["……"],
  "requirements": ["……"]
}
```

### `rules.writing_style`

- 类型：字符串
- 作用：描述整体文风
- 适合写什么：
  - 节奏快还是慢
  - 叙述偏现实还是偏戏剧化
  - 对话密度高还是描写密度高
  - 用词克制还是浓烈

示例：

```json
"writing_style": "偏现实向悬疑互动。信息逐步揭露，语气克制，重细节和对话，不要过度抒情。"
```

建议：

- 写“要像什么”，也写“不要像什么”
- 不要只写抽象形容词，比如“高级”“细腻”，最好补上行为描述
- 一段话就够，不要写成十几条口号

### `rules.forbidden`

- 类型：字符串数组
- 作用：明确系统禁止出现的内容或写法

示例：

```json
"forbidden": [
  "禁止角色突然性格反转且没有铺垫",
  "禁止开局立刻进入高强度亲密场景",
  "禁止AI跳出角色解释系统规则"
]
```

建议：

- 每条禁止事项尽量具体
- 禁止“剧情结果”，不如禁止“生成方式”
- 不要堆太多互相矛盾的禁令

### `rules.requirements`

- 类型：字符串数组
- 作用：写你希望 LLM 每轮尽量做到的事

示例：

```json
"requirements": [
  "每轮推进一点关系变化，不要原地打转",
  "每轮必须给玩家明确可执行的选择",
  "重要情绪变化必须体现在对白或动作里"
]
```

## 5. 数值维度 `dimensions`

维度是这个系统最核心的“剧情状态机接口”。事件、阶段、结局大多都靠它驱动。

单个维度结构：

```json
{
  "id": "trust",
  "name": "信任值",
  "range": [0, 100],
  "initial": [40, 60]
}
```

支持字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | 字符串 | 是 | 程序内部唯一标识 |
| `name` | 字符串 | 是 | 给 LLM 看的显示名 |
| `range` | 两元素数组 | 是 | 最小值和最大值 |
| `initial` | 两元素数组 | 强烈建议 | 初始范围 |
| `description` | 字符串 | 否 | 解释这个维度代表什么 |
| `scope` | 字符串 | 否 | 显示在该维度后面的说明，可用宏 |

### `id` 怎么写

要求：

- 只能用英文、数字、下划线、短横线
- 不能以数字开头
- 不能和其他维度重复

建议：

- 用英文短词：`trust`、`guilt`、`stress`
- 不要用中文、空格、复杂缩写

### `name` 怎么写

`name` 是 LLM 更容易识别的维度名，建议用中文：

```json
"name": "怀疑值"
```

### `range`

系统会自动把 LLM 输出 clamp 到范围内。

例如：

```json
"range": [0, 100]
```

建议：

- 大多数连续型情绪、风险、关系或调查进度类维度都适合用 `[0, 100]`
- 如果是离散程度更强的状态，也可以用小范围，例如 `[0, 10]`

### `initial`

支持固定值或区间随机，当前实现推荐写区间数组。

示例：

```json
"initial": [0, 10]
```

如果最小值和最大值相同，就等于固定值：

```json
"initial": [50, 50]
```

### `description` 和 `scope`

这两个字段主要帮助作者和模型理解维度含义。

示例：

```json
{
  "id": "heartbeat",
  "name": "警觉值",
  "description": "玩家对异常线索的警觉程度",
  "scope": "对当前目标",
  "range": [0, 100],
  "initial": [0, 0]
}
```

### 维度设计建议

推荐分三类来想：

1. 关系维度
   - 爱意值
   - 信任值
   - 依赖值

2. 风险维度
   - 怀疑值
   - 压力值
   - 愧疚值

3. 剧情推进维度
   - 心动值
   - 欲望值
   - 失控值

经验建议：

- 5 到 12 个维度通常最好用
- 太少，事件和阶段不够分层
- 太多，LLM 关注点会分散

## 6. 角色 `characters`

角色定义用于：

- 让 LLM 知道有哪些角色
- 提供宏引用
- 让开场生成和长期对话更稳定

单个角色结构：

```json
{
  "id": "captain",
  "name": "列车长",
  "description": "掌握部分关键信息，但不会主动把真相全盘托出。"
}
```

支持字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | 字符串 | 是 | 唯一标识 |
| `name` | 字符串 | 是 | 身份标签 / 展示名 |
| `description` | 字符串 | 建议 | 角色信息 |

### 必须包含 `player`

当前校验要求：

```json
{ "id": "player", "name": "玩家", "description": "由玩家代入的主角" }
```

如果没有这个角色，剧本无法通过校验。

### 关于角色 ID 和“内置角色”

严格来说，系统没有真正意义上的“内置角色模板”。除 `player` 会被校验要求存在之外，其他角色 ID 都是剧本作者自由定义的。

当前代码层面只额外提供了两个历史兼容宏别名：

| 宏 | 何时有意义 |
|----|------------|
| `{{partner}}` | 只有当你的剧本里真的存在 `id: "partner"` 的角色时 |
| `{{target}}` | 只有当你的剧本里真的存在 `id: "target"` 的角色时 |

这只是当前实现里的便捷别名，不代表系统推荐你必须使用这些角色命名，也不代表“现任 / 危险对象”是通用题材设定。

### 角色描述怎么写

建议每个角色描述包含：

- 身份
- 性格
- 与玩家关系
- 行为习惯
- 情绪触发点

不要只写：

```json
"description": "很帅，很有魅力"
```

这种信息密度太低，LLM 很容易把角色写成空壳。

## 7. 事件 `events`

事件是在“满足某些条件时”触发的剧情钩子。

单个事件结构：

```json
{
  "name": "列车长盘问",
  "stages": [1, 2],
  "trigger": {
    "suspicion": { "min": 70 }
  },
  "description": "列车长不再旁敲侧击，而是直接追问玩家刚才的异常行动",
  "once": true,
  "cooldown": 3
}
```

支持字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | 字符串 | 建议 | 事件名 |
| `stage` | 数字 | 否 | 单个阶段限制，兼容旧写法 |
| `stages` | 数字数组 | 否 | 多阶段限制，推荐新写法 |
| `trigger` | 对象 | 是 | 触发条件 |
| `description` | 字符串 | 建议 | 事件含义 |
| `once` | 布尔 | 否 | 是否只触发一次 |
| `maxTriggers` | 非负整数 | 否 | 最多触发次数 |
| `cooldown` | 非负整数 | 否 | 冷却轮数 |
| `effects` | 对象 | 否 | 持续效果 |

### 单阶段、多阶段、全阶段

当前版本支持三种写法：

单阶段：

```json
"stage": 1
```

多阶段：

```json
"stages": [1, 2, 3]
```

全阶段：

- 不写 `stage`
- 不写 `stages`

例如：

```json
{
  "name": "压力失眠",
  "trigger": { "stress": { "min": 80 } },
  "description": "只要压力足够高，任何阶段都可能发生"
}
```

### `trigger`

事件必须有触发条件，否则不会触发。

最简单写法：

```json
"trigger": {
  "suspicion": { "min": 50 }
}
```

复杂写法见后面的“条件系统”章节。

### `once`

如果事件只能发生一次：

```json
"once": true
```

### `maxTriggers`

如果事件可以重复，但有上限：

```json
"maxTriggers": 2
```

### `cooldown`

避免事件连续触发：

```json
"cooldown": 3
```

表示事件触发后，3 轮内不会再次触发。

### `effects`

事件触发后，还可以带来持续效果：

```json
"effects": {
  "sticky": {
    "dims": {
      "suspicion": 3,
      "stress": 2
    },
    "duration": 2
  }
}
```

表示：

- 事件触发后
- 接下来 2 轮
- 每轮额外让 `suspicion +3`、`stress +2`

### 事件设计建议

好的事件通常具备三个特征：

1. 条件明确
   - 不靠“模糊氛围”触发
   - 靠清晰维度阈值触发

2. 含义具体
   - 事件名和描述能让 LLM 理解“这是哪种剧情变化”

3. 不过度重复
   - 用 `once`、`cooldown`、`maxTriggers` 控制频率

## 8. 阶段 `stages`

阶段定义剧情所处的大状态。

单个阶段结构：

```json
{
  "name": "危险接触",
  "transition": {
    "heartbeat": { "min": 40 }
  }
}
```

支持字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | 字符串 | 建议 | 阶段名 |
| `transition` | 对象 | 除第一阶段外建议填写 | 进入下一阶段的条件 |

### 阶段推进规则

当前实现是：

- 当前阶段是 `0`
- 检查下一阶段的 `transition`
- 如果满足，就进入下一阶段
- 如果下一阶段也满足，会继续推进

也就是说，系统支持“一次数值变化连续跨多个阶段”。

### 阶段数组示例

```json
"stages": [
  { "name": "稳定生活" },
  { "name": "危险接触", "transition": { "heartbeat": { "min": 30 } } },
  { "name": "秘密关系", "transition": { "trust": { "max": 40 } } },
  { "name": "失控边缘", "transition": { "stress": { "min": 70 } } },
  { "name": "结局" }
]
```

建议：

- 第一阶段一般不写 `transition`
- 最后一阶段一般作为终局缓冲区，不一定非要有条件
- 阶段名要能反映关系状态，而不是只写“第一章、第二章”

## 9. 结局 `endings`

结局是全局终止条件。

单个结局结构：

```json
{
  "name": "BE-彻底翻车",
  "condition": {
    "suspicion": { "min": 100 }
  },
  "description": "真相彻底暴露，关系全面崩塌。"
}
```

支持字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | 字符串 | 建议 | 结局名 |
| `condition` | 对象 | 是 | 结局触发条件 |
| `description` | 字符串 | 建议 | 结局说明 |

### 结局检查方式

每轮输出后系统都会检查所有结局，命中第一个满足条件的结局后，会把会话标记为结束。

建议：

- 让不同结局使用不同的主导维度组合
- 不要把所有结局都写成“某维度 >= 100”
- 可以设计：
  - 高爱意低愧疚的 HE
  - 高怀疑高压力的 BE
  - 高依赖高占有的扭曲结局

## 10. 开局选项 `setup`

`setup` 决定玩家开始时需要回答哪些问题。

单个步骤结构：

```json
{
  "step": "你的职业",
  "description": "这会影响角色社交环境和日常场景",
  "options": [
    { "label": "设计师", "value": "设计师" },
    { "label": "律师", "value": "律师" }
  ],
  "allow_custom": true
}
```

支持字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `step` | 字符串 | 是 | 步骤名 |
| `description` | 字符串 | 否 | 给玩家看的补充说明 |
| `options` | 数组 | 建议 | 选项列表 |
| `allow_custom` | 布尔 | 否 | 是否允许自定义输入 |

单个选项结构：

```json
{
  "label": "异地恋",
  "value": "异地恋",
  "description": "两人不住在一起，见面频率低，误会更容易积累"
}
```

选项字段：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `label` | 字符串 | 是 | 显示文字 |
| `value` | 字符串 | 建议 | 实际写入 prompt 的值 |
| `description` | 字符串 | 否 | 帮助 LLM 理解这个选项的含义 |

说明：

- 当前实现支持 `description`
- 当前实现里如果 `value` 缺失，会默认回退成 `label`

## 11. 宏系统

当前宏替换实现支持以下内容：

| 宏 | 说明 |
|----|------|
| `{{player}}` | 玩家角色名 |
| `{{partner}}` | `id=partner` 角色名 |
| `{{target}}` | `id=target` 角色名 |
| `{{stage}}` | 当前阶段名称 |
| `{{stage_num}}` | 当前阶段编号，从 1 开始 |
| `{{dim:trust}}` | 维度当前值，支持 `id` 或 `name` |
| `{{dim:信任值}}` | 同上 |
| `{{random:A,B,C}}` | 随机选择一个 |

### 宏可以用在哪里

最实用的场景有：

- `rules.writing_style`
- `rules.forbidden`
- `rules.requirements`
- 角色描述
- 维度 `scope`

示例：

```json
"writing_style": "当前阶段是 {{stage}}。你要写出调查关系如何逐步紧张。"
```

```json
"scope": "对当前目标"
```

### 宏使用建议

- 用宏补“动态信息”，不要用宏拼复杂句子
- `{{dim:...}}` 更适合做提示，不适合承担完整规则
- `{{random:...}}` 可以做轻微变化，但不要依赖它承载核心逻辑

## 12. 条件系统

事件、阶段、结局都使用同一套条件格式。

### 12.1 旧格式：直接按维度写

```json
{
  "suspicion": { "min": 50 },
  "stress": { "min": 60 }
}
```

这表示 AND 逻辑，也就是“全部满足”。

### 12.2 新格式：叶子条件

```json
{
  "dim": "suspicion",
  "min": 50,
  "max": 80,
  "probability": 0.5
}
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `dim` | 字符串 | 维度 id |
| `min` | 数字 | 最小值（含） |
| `max` | 数字 | 最大值（含） |
| `probability` | 数字 | 0 到 1 的概率 |

### 12.3 组合条件

AND：

```json
{
  "op": "and",
  "conditions": [
    { "dim": "suspicion", "min": 50 },
    { "dim": "stress", "min": 60 }
  ]
}
```

OR：

```json
{
  "op": "or",
  "conditions": [
    { "dim": "guilt", "min": 70 },
    { "dim": "stress", "min": 80 }
  ]
}
```

NOT：

```json
{
  "op": "not",
  "conditions": [
    { "dim": "trust", "min": 60 }
  ]
}
```

### 条件设计建议

- 先用简单阈值跑通，再上复杂组合
- 概率条件适合给事件，不适合给阶段
- 阶段条件尽量清晰单向，避免前后互相打架

## 13. 校验规则和常见报错

当前校验器会重点检查这些内容：

### 必报错

- 不是 JSON 对象
- `characters` 里没有 `player`
- 维度 `id` 重复
- 维度 `id` 非法
- 维度缺少 `id`
- 维度缺少 `name`
- 维度 `range` 不是两个数字
- 事件阶段越界
- `stages` 不是数组或数组中有非法值
- `cooldown` / `maxTriggers` 不是非负整数
- `once` 不是布尔值
- 条件引用了不存在的维度
- 持续效果引用了不存在的维度

### 可能是警告

- 缺少剧本名称
- 某个维度 `initial` 不合法
- 事件同时写了 `stage` 和 `stages`
- 事件 `stages` 是空数组

### 最容易犯的错

1. 忘记写 `player`
2. 条件里引用的是维度中文名，但新格式 `dim` 其实应该优先用维度 `id`
3. 阶段索引从 0 开始，不是从 1 开始
4. 事件太多但维度太少，导致所有事件都挤在同一阈值附近

## 14. 写剧本的推荐流程

如果你不是先写完整 JSON，而是先设计玩法，建议按这个顺序来：

1. 先定题材和主冲突
   - 这个剧本的核心矛盾是什么
   - 玩家最常做的选择是什么

2. 再定角色
   - 至少写出 `player`
   - 主要角色分别掌握什么信息，彼此的冲突点是什么

3. 再定维度
   - 哪些值决定关系变化
   - 哪些值决定风险升级

4. 再定阶段
   - 关系是怎么一步步升级/恶化的

5. 再定事件
   - 每个阶段的代表性事件是什么

6. 最后补写作规则和结局
   - 把风格限制清楚
   - 把高概率终局条件定下来

## 16. 最后给作者的几条经验

- 先把“关系怎么变”写清楚，再写“文风怎么写”
- 事件少一点但更准，通常比事件很多但条件含糊更好
- 阶段名最好能表达关系状态，而不是流程编号
- 维度不要只是堆名词，要确保每个维度都真的会被事件、阶段或结局使用
- 如果导入后剧情不稳定，优先检查：
  - 维度是不是太多
  - 条件是不是过于复杂
  - 写作规则是不是互相打架
